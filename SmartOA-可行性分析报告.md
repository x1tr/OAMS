## 一、项目概述

SmartOA 是一个带 AI 智能问答的企业 OA 系统，基于微服务架构，涵盖员工管理、权限审批、公告发布、AI 客服四大核心模块。系统采用 Vue.js 3 + Spring Boot 微服务 + Ollama 本地大模型技术栈，面向企业内部日常办公场景。

### 核心目标

1. 员工可查公告、搜文档、提交请假申请、AI 问答
2. 管理者可审批下级申请、发布公告
3. 管理员可管理用户、角色、查看全系统数据
4. AI 客服基于知识库 RAG 检索增强生成，流式输出

### 2.2 架构可行性

**微服务拆分合理性**：

| 服务 | 职责单一性 | 数据独立性 | 可独立部署 | 评估 |
|------|-----------|-----------|-----------|------|
| Gateway | ✅ 仅路由+鉴权 | ✅ 无数据库 | ✅ | 合理 |
| oa-emp-service | ✅ 员工域 | ✅ 独立表 | ✅ | 合理 |
| oa-admin-service | ⚠️ 四合一 | ✅ 独立表 | ✅ | 可接受（实训简化） |
| oa-ai-service | ✅ AI域 | ✅ 独立 | ✅ | 合理 |

> **说明**：admin-service 承载 RBAC+审批+公告+日志四项功能，在正式生产环境中建议进一步拆分。但考虑到 10 天实训周期和 1 人后端的约束，当前聚合是务实的取舍。

**关键技术决策验证**：

| 决策 | 可行性 | 依据 |
|------|--------|------|
| AI 服务独立使用 Boot 3.4.5 | ✅ | Spring AI 1.0 强制要求 Boot 3.x，不能迁就主工程 Boot 2.x |
| 请假类型用自由文本 | ✅ | VARCHAR(100) 替代字典表，降低复杂度，适配性更强 |
| ES 同步用 try-catch 降级 | ✅ | ES 不可用时不影响主业务流程 |
| 内存向量库（无 RedisStack） | ✅ | JDK HashMap + 余弦相似度，0 额外依赖，8GB 内存足够 |
| 前端 SSE 流式接收 | ✅ | Fetch API + ReadableStream，浏览器原生支持 |

### 2.3 技术风险与应对

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| ES 内存不足闪退 | 中 | 高 | JVM 限制 512MB；提供 Docker/本地两种方案 |
| Ollama 模型下载慢 | 高 | 中 | 教师提前下载 qwen2.5:0.5b（400MB），U 盘拷贝分发 |
| Boot 2.x 与 3.x 混用 | 低 | 中 | AI 服务独立 POM，完全隔离 |
| 学生电脑配置不足 | 中 | 高 | 0.5b 小模型 + 内存向量库，8GB 可跑全套 |
| Nacos 单点故障 | 低 | 低 | 开发环境单机模式可接受 |

### 2.4 技术可行性结论：✅ 可行

> 所选技术栈均为业界成熟方案，有丰富文档和社区支持。风险点均已有明确应对策略，不存在阻塞性技术障碍。

### 3.3 后端单人工作量验证

```
emp-service:    3个Service方法 + 3条SQL + ES查询方法      ≈ 200行
admin-service:  6个Service方法 + 5条SQL + Redis缓存逻辑    ≈ 400行
ai-service:     2个Service方法 + RAG链路组装               ≈ 150行
───────────────────────────────────────────────────────────────
总计约 750 行业务代码，10 天平均 75 行/天，完全合理。
```

### 3.4 砍掉的功能（V1.0 不做）

| 功能 | 砍掉原因 |
|------|---------|
| 3 级以上复杂审批流 | 10 天做不完，只做员工→经理两级串行 |
| 数据大屏 / ECharts 图表看板 | 非核心功能 |
| 移动端适配 | 桌面端优先 |
| SSO / OAuth2.0 | 复杂度高，实训 JWT 足够 |
| 分布式事务 | 单体微服务无此需求 |
| CI/CD 流水线 | 实训无需自动化部署 |

### 3.5 进度可行性结论：✅ 可行（有前提）

> **前提条件**：教师课前完成所有骨架准备；学生提前 1 天完成环境安装；按天严格执行计划不拖延。每天留有的弹性时间可作为缓冲。
