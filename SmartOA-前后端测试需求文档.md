## 一、测试策略总纲

### 1.1 金字塔测试策略

```
           ╱  E2E 测试 ╲          ← 5%  核心流程端到端
          ╱  (5个场景)   ╲
         ╱───────────────╲
        ╱  集成/API 测试   ╲       ← 20% 接口联调 + 数据验证
       ╱   (所有API端点)    ╲
      ╱─────────────────────╲
     ╱    单元测试           ╲      ← 75% Service/Mapper/Vue组件
    ╱  (Service层 + 组件)    ╲
   ╱───────────────────────────╲
```

### 1.2 测试介入时间线

```
D1 ─── 编写测试用例 + 环境验证 + 中间件连通性测试
D2 ─── API接口测试（emp-service）+ 持续
D3 ─── 前端组件测试 + API联调验证
D4 ─── 权限测试 + 审批流程测试
D5 ─── 缓存验证 + 公告测试
D6 ─── AI服务测试（对话 + 知识库）
D7 ─── 流式渲染测试 + RAG链路验证
D8 ─── 全流程端到端测试
D9 ─── 回归测试 + Bug清单跟踪 + 验收测试
D10 ── 演示环境最终验证
```

> **核心原则**：测试从 D1 开始介入，不是最后才测。发现 Bug 立刻提给对应的人，不积累到 D9 集中爆雷。

### 1.3 测试环境

| 环境 | 用途 | 配置 |
|------|------|------|
| 本地开发环境 | 日常测试 | Windows 10/11, 8GB+ RAM |
| 中间件环境 | 连通性测试 | Docker / 本地安装 |
| 演示环境 | 最终验收 | 同开发环境（单机全栈部署） |

## 四、端到端（E2E）测试

### 4.1 核心场景（5个）

#### E2E-01：管理员全流程

```
登录(管理员) → 查看工作台统计 → 新增员工(输入完整信息) → 
ES搜索该员工 → 编辑员工信息 → 删除员工 → 发布公告 → 
ES搜索公告 → 查看操作日志
```
**验证点**：每个步骤的数据一致性，搜索结果的正确性。

#### E2E-02：自定义请假全流程

```
登录(普通员工) → 进入审批中心 → 新建请假申请 →
  填写：类型="考证"（自由文本）、时间="7.16-7.18"、事由="参加PMP认证考试"、
        附件=准考证.pdf →
  提交 → 查看我的申请(状态="待审批") →
退出 → 登录(部门经理) → 审批中心 → 待我审批 → 查看申请详情(验证：类型显示"考证") →
  点击"驳回" → 填写驳回理由"请补充培训机构证明" → 确认 →
  验证：申请状态变为"已驳回" →
登录(员工) → 查看申请详情 → 验证：驳回理由可见 → 修改后重新提交 →
登录(经理) → 点击"通过" → 验证：状态变为"已通过"
```
**验证点**：自由文本类型全程正确传递，驳回理由闭环，状态流转正确。

#### E2E-03：AI 客服 RAG 问答

```
登录(员工) → 进入AI对话页 → 验证：欢迎语 + 建议问题显示 →
  输入"请假需要什么材料？" → 发送 →
  验证：① 逐字流式渲染  ② 首字延迟 < 3秒  ③ 内容引用制度文档  ④ 末尾有来源引用 →
  追问"那年假呢？" → 验证：上下文关联 → 
  输入"今天天气" → 验证：诚实告知无相关信息 →
  新建对话 → 验证：上下文清空
```
**验证点**：流式渲染体验，RAG 检索准确性，多轮对话上下文。

#### E2E-04：权限边界测试

```
登录(普通员工) → 验证：侧栏无"系统设置" →
  直接访问 /settings  → 验证：重定向或无权限提示 →
  尝试调用 DELETE /api/emp/1  → 验证：403 拒绝 →
登录(管理员) → 验证：侧栏有"系统设置" → 访问用户管理 →
  修改普通员工角色为"部门经理" → 
登录(该员工) → 验证：现在可审批 + 菜单变化
```
**验证点**：前端路由守卫 + 后端 API 权限双重验证。

#### E2E-05：ES 降级测试

```
正常搜索"技术部" → 验证：返回 ES 结果 + 高亮 →
停掉 ES 服务 → 
再次搜索"技术部" → 验证：系统不报错，返回降级结果（MySQL LIKE 或空列表 + 提示） →
重启 ES → 恢复搜索 → 验证：搜索结果恢复
```
**验证点**：系统韧性，降级机制有效。

## 七、Bug 管理规范

### 7.1 Bug 严重程度

| 级别 | 定义 | 示例 | 处理时效 |
|------|------|------|---------|
| 🔴 P0-Critical | 系统阻塞，核心功能不可用 | 无法登录、AI 无响应、数据库连不上 | 立即修复 |
| 🟠 P1-Major | 核心功能异常，有 workaround | 搜索无高亮、缓存未命中 | 当天修复 |
| 🟡 P2-Minor | 非核心功能异常 | UI 偏移 2px、文字提示不准确 | D9 前修复 |
| 🟢 P3-Trivial | 建议改进 | 颜色可优化、文案可调整 | 有时间就改 |

### 7.2 Bug 提交流程

```
测试发现 Bug → 记录到 Bug 清单 → 判定严重程度 →

     │
     ▼
开发修复 → 测试验证 → ├── 通过 → 关闭
                       └── 不通过 → 重新打开 + 附复现步骤
```

## 十、关键测试风险

| 风险 | 应对 |
|------|------|
| 只有 1 名测试，无法覆盖全部 | 优先保证 P0 用例；开发自测 P1/P2；PM 协助 E2E |

| AI 回答质量不可预期 | 用小模型 + 限定知识库范围，质量可控；测试重点在链路打通而非内容完美 |
